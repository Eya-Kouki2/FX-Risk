Contexte & Objectifs
Problématique métier et réponse applicative
Contexte
📌 Problématique
Les établissements bancaires réalisent quotidiennement des opérations de change Spot FX avec des risques opérationnels élevés : montants importants, absences de références SWIFT, opérations hors-horaires, écarts de taux suspects. La traçabilité manuelle est insuffisante pour détecter ces anomalies en temps réel.
🎯 Objectif général
Fournir une plateforme web centralisée permettant de créer, contrôler, classifier, valider et surveiller les opérations FX, avec un workflow multi-rôles, un scoring automatique, des alertes, une cartographie et des prédictions.
✅ Valeur ajoutée
Réduction des risques opérationnels · Amélioration de la traçabilité · Accélération de la prise de décision · Supervision en temps réel · Audit complet des actions
Objectifs spécifiques
🔍
Détection automatique des anomalies
SWIFT manquant, montant élevé, écart de taux, hors-horaires
🏷️
Classification des risques
Scoring 0-100, niveaux : Low / Moderate / High / Critical
👥
Workflow multi-rôles
5 profils : Front Office, Back Office, Risk Team, Manager, Admin
📈
Supervision et reporting
Dashboard KRI, alertes, cartographie, exports PDF/CSV
🧠
Module prédictif
Projection heuristique du risque moyen sur 7 jours

🔐 Énumérations SQL
-- Rôles applicatifs CREATE TYPE app_role AS ENUM ( 'front_office', 'back_office', 'risk_team', 'manager', 'admin' ); -- Statuts opération CREATE TYPE operation_status AS ENUM ( 'draft', 'pending', 'validated', 'rejected', 'escalated', 'settled' ); -- Niveaux de risque CREATE TYPE risk_level AS ENUM ( 'low', 'moderate', 'high', 'critical' );
⚙️ Triggers automatiques
🎯 trg_operations_risk
Calcule automatiquement risk_score et risk_level à chaque INSERT ou UPDATE de l'opération (montant, taux, SWIFT, horaire, paire).
🔔 trg_operations_alerts
Génère les alertes dans la table alerts immédiatement après insertion d'une opération selon les seuils définis.
👤 trg_handle_new_user
Crée automatiquement le profil et attribue le rôle lors de l'inscription d'un nouvel utilisateur.
🕐 trg_..._updated
Met à jour updated_at automatiquement sur profiles et operations.
🔒 Row Level Security (RLS)
Toutes les tables sont protégées par des politiques RLS Supabase. Chaque rôle ne peut voir et modifier que ce que ses droits autorisent (ex : Front Office ne peut créer, Back Office valide, Risk Team accède aux alertes, etc.).

Moteur de Scoring Automatique
Trigger PostgreSQL compute_risk() — calcul du risk_score 0–100
Scoring Engine
⚖️ Règles de scoring (critères additifs)
+30
Format SWIFT invalide
Référence ne correspond pas au regex BIC ^[A-Z]{6}[A-Z0-9]{2,5}$
+25
Référence SWIFT absente
Champ NULL ou vide → risque de contrepartie élevé
+20
Montant > 1 000 000
Transaction de grande taille → exposition accrue
+20
Écart de taux > 2%
|taux_saisi - taux_marché| / taux_marché > 0.02
+15
Hors horaires (avant 7h / après 20h)
Opération créée en dehors des heures normales de trading
+10
Paire de devises inhabituelle
Non listée parmi : EUR/USD, USD/JPY, GBP/USD, USD/CHF…

⚡ Realtime — useRealtimeInvalidate
// Hook personnalisé : abonnement Supabase WebSocket // → invalide le cache React Query à chaque changement DB useRealtimeInvalidate("operations", [ ["ops-dash"], ["ops-total-count"] ]); useRealtimeInvalidate("alerts", [["alerts-dash"]]);
Toutes les pages s'actualisent automatiquement sans polling grâce aux subscriptions Supabase Realtime.
🔐 Authentification & Autorisation
-- Fonction SQL get_user_role() SELECT role FROM user_roles WHERE user_id = _user_id ORDER BY CASE role WHEN 'admin' THEN 1 WHEN 'manager' THEN 2 WHEN 'risk_team' THEN 3 END LIMIT 1
JWT Supabase + has_role() vérifié dans chaque policy RLS. Le front lit le rôle depuis la session pour adapter l'UI.
📈 Régression linéaire prédictive
// Calcul de la pente (slope) de l'historique 14j function regressionSlope(values: number[]) { const meanX = (n - 1) / 2; const meanY = average(values); const num = values.reduce( (sum, v, i) => sum + (i - meanX)*(v - meanY), 0); return denom ? num / denom : 0; } // Projection : risk_prévu = avg + slope × jours

💡 Points techniques à valoriser
🔧 Scoring côté base de données
Le calcul du risk_score est entièrement fait par un trigger PostgreSQL, garantissant cohérence et sécurité indépendamment du client.
⚡ Temps réel sans polling
Utilisation de Supabase Realtime WebSocket via un hook React personnalisé useRealtimeInvalidate pour invalider le cache automatiquement.
🔐 Sécurité à plusieurs niveaux
RLS PostgreSQL + JWT Supabase + logique UI par rôle. La sécurité est appliquée au niveau de la base de données, pas seulement de l'interface.
🧠 Module prédictif heuristique
Première version basée sur la régression linéaire (slope). Présentable comme fondation pour un futur modèle ML (LSTM, ARIMA…).
